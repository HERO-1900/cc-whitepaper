Harness Engineering
从 Claude Code 看 Agent 工程化范式

当 AI 模型像厨师，Harness 就是整个厨房——炉灶、刀具、操作规程、出菜标准。
探索 Agent 工程化从"发短信叫外卖"到"赛车安全带+仪表盘"的五代演化。

01 · 引子

为什么要多讲一章

前 82 章，我们逐行拆解了 Claude Code 的源码机制——从 agent loop 到 context compaction，从 permission system 到子 agent 防火墙。但当我们把视野拉远，会注意到行业里正在发生一个有趣的现象："Harness Engineering" 这个词开始频繁出现在技术博客、论文标题和会议演讲中。

更关键的是，Claude Code 是被反复引用的核心案例。LangChain 的 Vivek Trivedy 说 "If you're not the model, you're the harness"；OpenAI 的 Ryan Lopopolo 用 Claude Code 的 CLAUDE.md 来论证 AGENTS.md 的设计哲学；Anthropic 自己的 Justin Young 和 Nishaanth Rajasekaran 都在 harness 主题文章中以 Claude Code 为标杆。

三个必须单独成章的理由
  1. 名字本身就是知识——当行业为某种实践命名时，意味着它从"隐性经验"升级为"显性范式"
  2. Claude Code 是当事人——它是被引用最频繁的 harness 案例之一，白皮书不能缺席这场对话
  3. 影响你应该花多少工程精力在 harness 层——这不是纯理论，而是直接影响你今晚写不写那条 lint 规则的决策依据
💡

通俗理解：厨师与厨房

AI 模型像一位顶级厨师，拥有精湛的烹饪技艺。但仅有厨师无法经营一家餐厅——你需要厨房（炉灶、刀具、冰箱）、操作规程（先备菜再下锅、生熟分开）、出菜标准（摆盘规范、温度要求）和质量检验（试味、顾客反馈）。Harness 就是这套"厨房系统"，它本身不产生菜肴，但决定了厨师能否发挥全部实力。

03 · 演化

五代范式演化

Agent 工程化并非一夜之间的跃迁，而是经历了五个清晰的阶段。每一代范式都建立在上一代的基础之上，同时引入新的抽象层级：

第 1 代
~2022

Pure Prompt

写 prompt → 发送 → 接收 → 结束。无工具调用、无循环机制、无状态管理。这是 LLM 应用的原始形态，就像直接与模型对话，没有任何中间层。

💡 通俗类比：发短信叫外卖，对方只回一条消息就结束对话
第 2 代
2023

Tool Use

引入 Function Calling、MCP 工具调用。模型可以"动手"了——查天气、调 API、操作数据库。但仍是无状态的单轮交互，每次调用都是独立的。

💡 通俗类比：外卖平台接入了地图、库存、支付，但还是无状态单轮
第 3 代
2023-24

Multi-Agent / Orchestration

CrewAI、AutoGen、LangGraph 兴起。工程师开始设计 agent 之间的协作关系——谁负责规划、谁负责执行、谁负责验证。但流程结构仍需工程师事先"画好"。

💡 通俗类比：物流公司有调度员/骑手/客服，但流程靠工程师事先画好
第 4 代
2025 至今

Harness ← 当前位置

范式核心发生转移：不再只是设计 agent 做什么，而是精心设计 agent 运行的环境本身——进度文件、特性清单、启动脚本、lint 规则、反馈循环、垃圾回收……Harness 是 agent 的"安全带+仪表盘+赛道规则"。

💡 通俗类比：赛车的安全带 + 仪表盘 + 赛道规则
代表实践
• Justin Young 的 initializer + coding agent 双阶段架构
• Ryan Lopopolo 的 AGENTS.md-as-table-of-contents
• LangChain 的 generator-evaluator 反馈循环
第 5 代
争议中

端到端 RL

Noam Brown 等研究者认为 harness 只是"打补丁"，终将被更强的模型自然消灭。但 Rajasekaran 反驳："the space of interesting harness combinations doesn't shrink as models improve. Instead, it moves."（有趣的 harness 组合空间不会随着模型改进而缩小，它只是转移了位置。）

💡 通俗类比：自动驾驶从"辅助驾驶"到"完全 autonomous"的争议

05 · 学术

四篇学术论文速览

学术界对 Harness 的研究正在加速。以下四篇论文代表了该领域的前沿进展：

arXiv 2507.11633

Zhang 等 · UCSD / UCSB · 2025-07-15
关键发现：感知/记忆/推理三模块分离的 harness 在 Sokoban、2048、Candy Crush、Tetris 上相比 baseline 提升显著（p<0.05）；Candy Crush 中位分数 +217.50 分
贡献：证明 harness 组件对 agent 性能有可测量、可分解的因果影响
arXiv 2603.03329

06 · 争议

行业争议：Big Model vs Big Harness

Harness Engineering 并非没有争议。业界对此存在两种截然不同的立场：

⚠️
怀疑阵营
  • Chayenne Zhao 的"新名旧物"批评：每隔几个月就新造一个词，同一件事换不同包装，本质没有进化
  • Noam Brown 的 Bitter Lesson 立场：Harness 是"在 pre-training 不够强时打的补丁"，终将被更强的模型自然消灭
  • METR 研究：专业 harness 相比 basic scaffold 的优势 negligible（可忽略）
  • Scale AI SWE-Atlas：Harness 选择产生的性能差异在实验误差范围内
  • HumanLayer Kyle 的发现：Opus 4.6 在 Claude Code 官方 harness 下排名 #33，在 LangChain 重设的 harness 下排名 #5——可能是「模型对自己 harness 过拟合」
VS
✓
支持阵营
  • 分层是软件工程的基本方法：OS 在应用层之下，没人认为分层是"拐杖"
  • Rajasekaran 的实证：从 Opus 4.5 升级到 4.6 时，sprint 结构被简化、context reset 被移除，但 planner 和 evaluator 保留。"the space of interesting harness combinations doesn't shrink as models improve. Instead, it moves."
  • NLAH 的发现：Harness 作用区域在"模型差一点但做不到"的边界——边界往外移，harness 在新边界继续有价值
  • LangChain 实证：只改 harness 让同一模型在 Terminal Bench 2.0 提升 13.7 百分点
"You can engineer your way above the Bitter Lesson,
and harnesses can survive even reasoning paradigm changes."
— swyx，对 "Big Model vs Big Harness" 的经典总结
作者判断
Harness Engineering 是真实工程需求的系统化表达，不是全新发明，但值得独立命名。它是成熟软件工程原则——分层、约束、反馈循环、渐进增量——在 LLM agent 场景的具体化。与其争论它是不是"旧酒新瓶"，不如问：你的团队今天有没有在实践这些原则？有没有给它们应有的工程投入？

08 · 行动

给读者的五条行动建议

读完本章，你可以立即在自己的项目中实践以下原则：

1

先让 agent 失败，再添加 harness 组件

不要"预防性"过度设计。从最简单的情况开始，当 agent 真的出错时，再针对性添加约束或检查。这是避免 harness 膨胀的最佳实践。

2

用"load-bearing test"检验 harness 组件

逐一移除 harness 的每个组件，观察是否影响结果。如果移除后一切正常，那个组件就是"装饰性"而非"结构性"的，应该考虑删除。

3

CLAUDE.md / AGENTS.md 是目录，不是手册

遵循 Lopopolo 的约 100 行原则：文档应该像书的目录，指引方向而非罗列细节。细节应该放在代码注释或专门的文档中。

4

反馈循环比正向指令更有效

"可验证的失败比不可测量的成功更有价值。"设计让 agent 能快速试错并获得明确反馈的机制，而不是编写冗长的"应该如何做"指令。

5

Harness 有模型特异性，移植要重新 calibrate

不要假设为一个模型设计的 harness 能直接套用到另一个模型。HumanLayer 的发现（Opus 在 CC harness 下排名 #33，在 LangChain harness 下排名 #5）就是最好的警示。

Harness Engineering 不是未来时，而是现在进行时。Claude Code 的前 82 章源码已经展示了如何构建一个工业级的 agent harness；本章则试图回答"为什么这样设计"以及"行业正在如何演进"。
