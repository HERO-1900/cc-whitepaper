配置即代码

Configuration as Code — Imperative vs Declarative Paradigms

🎯
命令式 Imperative
"怎么做" How to do it
比喻 Metaphor
像GPS导航的"左转→直走200米→右转"
Like GPS turn-by-turn directions
# 一步一步告诉系统做什么 grep "pattern" file.txt read config.json edit line 42 "old" → "new" bash npm test
  • 精确控制每一步 Granular control
  • 脆弱 — 中间失败则全部失败 Brittle
  • 不可复用 — 特定于场景 Not reusable
VS
控制粒度 Control
精细 Fine
→
粗粒度 Coarse
容错性 Resilience
弱 Weak
→
强 Strong ✓
复用性 Reusability
低 Low
→
高 High ✓
学习成本 Learning
低 Low
→
中 Medium
灵活性 Flexibility
高 High
→
受系统限制 Limited
✨
声明式 Declarative
"要什么" What you want
比喻 Metaphor
像说"我要去机场" — 系统自己选最佳路线
Like saying "take me to the airport"
// CLAUDE.md — 声明期望状态 { "rules": ["使用TypeScript严格模式"], "testCoverage": ">80%" } // settings.json { "allowedTools": ["Read", "Edit"], "model": "claude-sonnet-4-6" }
  • 描述目标状态 Describe target state
  • 系统自动推导路径 Auto-deduce path
  • 容错强、可复用 Resilient & reusable
Claude Code 中的声明式配置实例
📋
CLAUDE.md
项目级声明 · Project-level
"always": "使用 TypeScript 严格模式"
"test": "覆盖率必须 > 80%"
"style": "优先函数式编程"
⚙️
settings.json
用户级声明 · User-level
"allowedTools": ["Read", "Edit"]
"permissionMode": "acceptEdits"
"model": "claude-sonnet-4-6"
🏢
managed-settings.json
企业级声明 · Enterprise-level
"requiredApprovals": 2
"blockedTools": ["Bash"]
"auditLevel": "full"
🔌
.mcp.json
集成声明 · Integration
"servers": [{
  "name": "github",
  "command": "npx -y @modelcontextprotocol/..."
}]
声明式配置层级 · Declarative Config Hierarchy
企业 Enterprise → 用户 User → 项目 Project → 命令行 CLI
后加载的配置覆盖前者 · Later configs override earlier ones
