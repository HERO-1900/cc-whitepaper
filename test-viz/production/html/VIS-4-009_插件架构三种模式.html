扩展性设计模式

插件架构的三种模式对比：Hooks · MCP · Plugins
Three Extension Patterns: Event-driven, Protocol-driven, Package-driven

Hooks Event-Driven / 事件驱动
运行机制 Mechanism

在 27个生命周期事件 上注册 shell 命令

settings.json / hooks
数据流向 Data Flow

事件触发 → stdin 传入 JSON → 执行脚本 → stdout 返回结果

优劣势 Pros & Cons
  • ✓ 零依赖、任何语言、最简单
  • ✗ 仅限事件触发、无状态、不能添加新工具
适用场景 Use Cases

自动格式化、通知推送、日志记录、简单自动化脚本

MCP Protocol-Driven / 协议驱动
运行机制 Mechanism

通过 JSON-RPC 2.0 与外部 MCP 服务器通信

.mcp.json
数据流向 Data Flow

Claude Code → MCP 客户端 → Transport → MCP 服务器 → 返回工具/资源/提示

优劣势 Pros & Cons
  • ✓ 标准协议、跨语言、可提供新工具和资源
  • ✗ 需要运行外部服务器、配置相对复杂
适用场景 Use Cases

数据库访问、API 集成、专业工具扩展、跨平台服务对接

Plugins Package-Driven / 包驱动
运行机制 Mechanism

加载 NPM 包，注入 tools 与 slash commands

package.json / claudeCodePlugin
数据流向 Data Flow

启动时扫描 → require 包 → 注册工具+命令 → 运行时调用

优劣势 Pros & Cons
  • ✓ 最强扩展性、可改变核心行为、类型安全
  • ✗ 仅 TypeScript/JavaScript、安全风险高
适用场景 Use Cases

深度定制、企业集成、自定义 UI 组件、核心行为改造

维度 Dimension Hooks — 事件驱动 MCP — 协议驱动 Plugins — 包驱动
复杂度 Complexity 低 — 单文件脚本即可 中 — 需部署/维护外部服务 高 — 完整 NPM 包开发流程
能力范围 Capability 事件响应，无状态执行 提供新工具、资源、提示 可修改核心行为，UI 级扩展
安全性 Security 最高 — 隔离执行，沙箱shell 中等 — 网络边界约束 较低 — 代码注入风险
典型用例 Typical Use 自动格式化、通知、日志 数据库、API、专业工具 企业集成、深度定制
Hooks(轻度) ⊂ MCP(中度) ⊂ Plugins(重度)  |  三者互补，共同构成完整的扩展生态
