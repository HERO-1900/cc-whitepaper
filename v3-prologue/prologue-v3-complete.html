§ 4

为什么用「操作系统」来理解 Claude Code

在进入代码细节之前，我们需要建立一个整体框架。这个框架不是为了让描述听起来更宏大——它是一张工作地图，帮助你在阅读后续章节时，始终知道自己在整个系统的哪个位置。

4.1 操作系统映射

将 Claude Code 与操作系统类比，有两个核心同构点值得精确标注：

OS 概念 Claude Code 实现
Kernel（内核） QueryEngine —— 进程调度器 · Prompt 组装 · 工具分发
负责将用户请求分解为可执行的工具调用序列，管理调用状态机
Syscall Interface（系统调用） 40 个工具 —— 文件 / 进程 / 网络 / 子 Agent 的统一接口
每个工具有严格的 JSON Schema 定义，是 AI 与真实系统交互的唯一合法通道
💡 通俗理解

QueryEngine 就像一条快递分拣流水线：输入是一张「收件单」（用户请求），输出是一系列「派送任务」（工具调用）。流水线本身不送货，但它决定每个包裹走哪条路、优先级是什么、出错了如何重试。工具就像员工的「持证上岗」证书——没有这张证，AI 根本无法触碰文件系统或执行命令。

4.3 安全防线：多层防御架构

对一个拥有文件读写、Shell 执行、网络访问能力的 AI Agent 而言，安全不是可选项——它是系统能够存在的前提条件。Claude Code 实现了 5 层递进式防御，每一层独立生效且不可被单独绕过：

Layer 1
权限模型
bypassPermissionsKillswitch 全局开关 + 分级权限白名单。98 个斜杠命令各自携带独立的权限声明，任何越权操作在路由阶段即被拦截。
Layer 2
工具调用沙箱
每次工具调用前进行参数验证（JSON Schema 校验 + 路径规范化），防止路径穿越、命令注入等常见攻击向量。
Layer 3
Prompt Injection 防御
恶意内容尝试通过文件读取或外部数据注入指令时，系统维护独立的「指令来源」追踪，用户指令与工具返回内容保持严格隔离。
Layer 4
互斥锁设计
关键操作（如文件批量修改、Shell 脚本执行）实现互斥调度，防止并发操作下的竞态条件导致不可预测的系统状态。
Layer 5
Fail-Closed 设计
任何权限判断的异常路径默认走「拒绝」而非「允许」。即使权限系统本身出现 bug，也不会意外放行危险操作——这是反脆弱设计的核心体现。
💡 通俗理解

多层防御就像医院手术室的进入流程：消毒→换衣→戴手套→进无菌台，每一层独立设计，且不可跳过任何一层。即使在其中某一层出现操作失误，其他层仍然有效。Claude Code 的安全模型遵循同样的逻辑——单点失效不会导致整体崩溃。

4.4 外部接口：Hook 事件与 MCP 连接

Claude Code 通过两套机制向外部世界开放扩展能力：27 个 Hook 事件节点，让开发者可以在 Agent 生命周期的任意阶段注入自定义逻辑；8 种 MCP 传输规格，让 Claude 能够连接任意外部数据源和服务。

MCP（Model Context Protocol）是 Claude Code 连接外部世界的标准化接口。通过 MCP，开发者可以将数据库、API、文档系统等任意外部服务接入 Claude 的工具体系，而无需修改 Claude Code 本身的代码：

💡 通俗理解

Hook 像「快递柜的短信通知」——快递到了（事件触发），你收到消息，可以选择去取（执行自定义逻辑），也可以忽略，快递柜仍然正常运作。MCP 像「MacBook 的 USB-C 转接口」——MacBook 本身不关心你接的是显示器还是硬盘，转接口提供统一的物理接口，外设自己实现各自的逻辑。
