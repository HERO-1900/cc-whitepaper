主题切换 PoC

本沙箱演示 CC 白皮书的双主题方案：CSS 变量分层 + 三态切换器（系统/亮/暗）+ localStorage 记忆 + FOUC 防御。所有颜色来自 theme-tokens.css， 页面本身不写一个硬编码色值。

当前状态 → data-theme = ? localStorage = ? 系统偏好 = ?

设计要点

why 三层结构

原子层（primitive）只声明颜色值，语义层（color-bg-*）声明用途，组件层 只引用语义层。这样换主题时只需要重映射语义→原子的指针，不需要碰组件代码。 未来如果要加"高对比度模式"或"打印模式"，新增一个 [data-theme="hc"] 块即可，组件零改动。

切换优先级

用户手动选择 (localStorage)
   ↓ 没有则
系统偏好 (prefers-color-scheme)
   ↓ 没有则
默认 dark（与生产站点一致）

FOUC 防御原理

把 localStorage.getItem('cc-theme') 放在 <head> 的同步 <script> 里，浏览器在解析 CSS 之前就把 data-theme 设到 <html> 上。 CSS 第一次匹配选择器时已经是正确的主题，因此不会出现"先渲染暗色再切亮色" 的闪烁。代价是 5 行同步 JS 阻塞渲染，可以接受。

通俗理解

Token 系统就像菜谱里的"一勺糖"——具体多少克是另一回事。组件只说"加一勺 --color-accent"，到底是紫还是蓝，全靠主题文件这本菜谱定义。

正文示例

Claude Code 的核心循环是 nO 函数，它接收消息流并产出工具调用决策。 在 30 万行源代码中，nO 只占不到 200 行，但承担了整个 Agent 循环的调度职责。 这种"小函数承担大责任"的模式让代码审计变得相对容易。

对比 Cursor 或 Cline，CC 的差异在于：它把权限决策从运行时抽到了编译时—— 通过 CacheSafeParams 类型把不安全的参数组合直接拒绝在类型检查阶段。 这是一个非常 Anthropic 风格的工程决策。
